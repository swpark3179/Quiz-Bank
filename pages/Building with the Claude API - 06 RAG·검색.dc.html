① 시작·개요② API 접근③ 프롬프트 평가④ 엔지니어링⑤ 도구 사용⑥ RAG·검색⑦ Claude 기능⑧ MCP⑨ Anthropic 앱⑩ 에이전트

RAG(검색 증강 생성) = 언어 모델로 대용량 문서를 질의하는 기법. 컨텍스트 한계에 부딪히지 않고 100~1000+ 페이지에서 특정 정보를 추출한다.

01

RAG 소개

Introducing RAG
Option 1 · 직접
전체 문서를 프롬프트에

토큰 한계, 긴 프롬프트로 효과↓, 비용↑, 느림.

Option 2 · RAG
관련 청크만 찾아 포함

문서를 작은 청크로 분할 → 질문마다 관련 청크만. 확장·저비용·빠름.

핵심 과제

RAG는 단순성을 내주고 확장성·효율을 얻지만, 사용 사례에 맞는 관련성 정의와 최적 청킹 전략이라는 신중한 구현·평가가 필요하다(청크가 완전한 맥락을 담는다는 보장 없음).

02

텍스트 청킹 전략

Text chunking strategies

청킹 품질이 RAG 성능을 좌우한다. 나쁜 청킹은 무관한 컨텍스트 검색을 부른다(소프트웨어 "bug" 질의에 의학 "bug" 검색).

크기 기반
같은 길이로 분할. 가장 흔함 / 단어 잘림 → 오버랩으로 보완.
구조 기반
헤더·문단·섹션으로 분할. 마크다운/HTML에 최적(예: ##).
시맨틱 기반
NLP로 의미 유사 문장 묶음. 가장 고급, 구현 복잡.
규칙

보편적 최선의 청킹법은 없다. 문자 단위(신뢰할 폴백) · 문장 단위(좋은 중간 지점) · 섹션 단위(최적이나 구조화 입력 필요) — 문서 구조와 사용 사례로 결정.

03

텍스트 임베딩

Text embeddings

텍스트 임베딩 = 텍스트 의미의 수치 표현. 임베딩 모델이 텍스트 → 긴 숫자 리스트(-1 ~ +1) 출력. 각 숫자는 텍스트의 미지의 특성을 점수화한다.

  • 시맨틱 검색 = 임베딩으로 질문과 관련된 청크를 찾기 → RAG의 검색 문제(질의↔청크 매칭) 해결.
  • 파이프라인: 청크 추출 → 질의 → 시맨틱 검색 → 관련 청크를 프롬프트 컨텍스트로 추가.
  • 구현: Anthropic은 임베딩 생성에 Voyage AI 권장(별도 계정/키, 무료 시작, SDK 통합).
핵심

임베딩은 키워드 매칭이 아닌 의미 유사도 매칭을 가능케 해 더 나은 텍스트 관계 이해를 제공한다.

04

전체 RAG 흐름

The full RAG flow
  1. 텍스트 청킹 — 소스 문서를 조각으로 분할
  2. 임베딩 생성 — 청크를 수치 벡터로 변환
  3. 정규화 — 벡터 크기를 1.0으로 스케일(API가 자동 처리)
  4. 벡터 DB 저장 — 벡터 연산에 최적화된 DB에 저장
  5. 질의 처리 — 사용자 질문을 같은 모델로 임베딩
  6. 유사도 검색 — 코사인 유사도로 가장 유사한 임베딩 찾기
  7. 프롬프트 조립 — 질문 + 검색된 청크를 LLM에 전송
핵심 수학

코사인 유사도 = 벡터 사이 각도의 코사인(-1~1, 1에 가까울수록 유사). 코사인 거리 = 1 − 유사도(0에 가까울수록 유사). 전처리(1~4) → 질의 → 실시간 검색(5~7) → LLM 응답.

05

RAG 흐름 구현하기

Implementing the RAG flow
  1. 청킹 — report.md에 chunk_by_section으로 섹션 분할
  2. 임베딩 — generate_embedding으로 각 청크 벡터 생성
  3. 벡터 스토어 채우기 — zip()으로 청크-임베딩 쌍 순회, store.add_vector(embedding, {content: chunk})
  4. 질의 처리 — 사용자 질문을 임베딩
  5. 유사도 검색 — store.search(user_embedding, 2)로 관련 청크 2개(코사인 거리 예: 0.71, 0.72)
메타데이터 저장

원본 텍스트를 임베딩과 함께 저장해야 의미 있는 검색 결과를 얻는다. 이 워크플로우는 완성됐지만 한계가 있어 BM25·멀티 인덱스·리랭킹으로 개선한다.

06

BM25 렉시컬 검색

BM25 lexical search

BM25(Best Match 25) = 시맨틱 검색을 보완하는 렉시컬(어휘) 검색. 시맨틱만으론 정확한 용어 매칭을 놓칠 수 있다. 하이브리드 검색 = 시맨틱(벡터) + 렉시컬(BM25)을 병렬 결합 후 병합.

  1. 질의를 개별 용어로 토큰화(구두점 제거, 공백 분할)
  2. 모든 청크에서 각 용어 빈도 카운트
  3. 사용 빈도 기반 중요도 부여(희귀 용어 = 높은 중요도, "a" 같은 흔한 용어 = 낮음)
  4. 높은 가중치 용어를 자주 포함하는 청크 순으로 랭킹

BM25는 정확한 용어 매칭에 강해 시맨틱의 약점을 보완. 둘 다 유사한 API(add_document, search)라 결합이 쉽다.

07

멀티 인덱스 RAG 파이프라인

A Multi-Index RAG pipeline

시맨틱(벡터 인덱스)과 렉시컬(BM25 인덱스)을 Retriever 클래스로 결합해 두 인덱스 결과를 병합한다.

Reciprocal Rank Fusion (RRF)

# 각 문서에 대해 모든 검색 방법에 걸쳐 합산 RRF_score = Σ ( 1 / (rank + 1) ) # 예 벡터: [doc2, doc7, doc6] BM25: [doc6, doc2, doc7] 최종: [doc2, doc6, doc7] # doc2가 양쪽에서 높음
이점

서로 다른 검색 패러다임 결합으로 정확도↑, 표준 API(search()·add_document())의 모듈식 설계로 인덱스 추가 확장이 쉽고 단일 방법 실패 케이스를 더 잘 처리한다.

결과 리랭킹

Reranking results

리랭킹 = 초기 검색 후 LLM으로 결과를 관련성 순으로 재정렬하는 후처리 단계.

→

트레이드오프

효율을 위해 전체 텍스트 대신 문서 ID 사용 + 프리필/정지 시퀀스로 구조화 JSON. 정확도↑ ↔ 추가 LLM 호출로 지연↑. 초기 검색이 뉘앙스를 놓칠 때 효과적(예: "ENG team" vs "engineering team").

09

컨텍스추얼 검색

Contextual retrieval

컨텍스추얼 검색 = 임베딩 전에 각 청크에 맥락을 추가해 정확도를 높이는 기법. 청크가 원본의 맥락을 잃는 문제를 해결한다.

  1. 개별 청크 + 원본 소스 문서 준비
  2. LLM(Claude)에 "이 청크의 맥락을 생성하라" 전송
  3. LLM이 청크와 더 큰 문서의 관계를 설명하는 짧은 맥락 생성
  4. 생성 맥락 + 원본 청크 결합 = "맥락화된 청크"
  5. 맥락화된 청크를 벡터/BM25 인덱스 입력으로 사용
대용량 문서 전략

시작부 스타터 청크(1~3개, 요약용) + 대상 청크 바로 앞 청크(국소 맥락) 포함, 관련성 낮은 중간 청크는 건너뜀. add_context 함수가 맥락화 버전을 반환한다.
