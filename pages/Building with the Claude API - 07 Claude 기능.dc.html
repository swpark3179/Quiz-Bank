01

확장 사고

Extended thinking

최종 응답 생성 전 추론 시간을 갖게 하는 기능. 사용자에게 별도의 사고 과정이 보이며, 복잡한 작업의 정확도↑(단 비용·지연↑).

  • 사고 예산(thinking budget) = 사고 단계에 할당하는 최소 1024 토큰. max_tokens는 사고 예산보다 커야 함(예산 1024 → max_tokens ≥ 1025).
  • 언제: 프롬프트 최적화로도 정확도가 안 나올 때. prompt eval로 필요성 판단.
  • 구현: thinking=true + thinking_budget 설정, max_tokens > thinking_budget 보장.

02

이미지 지원

Image support

Claude Vision = 사용자 메시지 내 이미지를 분석·비교·카운트·기술. 요청당 최대 100개 이미지(크기·치수 제한), 토큰 소비(픽셀 높이×너비 기반).

  • 이미지 블록: 원시 데이터(base64) 또는 URL 참조. 구조: type: image, source, media_type, data + 프롬프트 텍스트 블록.
정확도의 핵심 = 프롬프트

⚠️ 강력한 프롬프팅이 필수 — 단순 프롬프트는 자주 실패. 단계별 분석 지침, one/multi-shot 예시, 검증 단계를 쓴다. 이미지 정확도는 이미지 품질이 아니라 프롬프트 정교함에 달려 있다(예: 위성 이미지 화재 위험 평가).

03

PDF 지원

PDF support

Claude는 이미지 처리와 유사한 코드로 PDF를 직접 읽는다(텍스트+이미지+차트+표 추출).

항목이미지PDF
파일 타입"image""document"
미디어 타입"image/png""application/pdf"
변수명image_bytesfile_bytes

사용 패턴은 이미지 입력과 동일하되 document 전용 파라미터를 쓴다 — 종합 문서 분석의 원스톱 솔루션.

04

인용

Citations

인용 = Claude가 출처 문서를 참조하고 정보의 출처를 보여 주는 기능.

  • citation_page_location — PDF용(문서 인덱스/제목/시작·끝 페이지/인용 텍스트)
  • citation_char_location — 평문용(텍스트 블록 내 문자 위치)
  • 구현: 요청에 "citations": {"enabled": true} + 출처 식별용 "title". PDF·평문 모두 지원.
UI 이점

참조 콘텐츠 호버 시 출처 문서·페이지·정확한 인용 텍스트를 보여 주는 팝업 구현 가능 → Claude가 "기억에서 말하는" 게 아니라 출처에서 응답을 구성함을 보장한다.

05

프롬프트 캐싱

Prompt caching

프롬프트 캐싱 = 이전 요청의 계산 작업을 재사용해 응답을 빠르게 하고 비용을 줄이는 기능.

일반 요청은 입력 처리(내부 자료구조 생성·계산) → 출력 → 모든 처리 작업 폐기한다. 후속 요청이 같은 입력을 담으면 방금 버린 계산을 다시 반복하는 비효율이 발생한다.

해법

입력 처리 결과를 폐기하지 않고 임시 캐시에 저장 → 동일 입력이 다시 오면 재처리 대신 캐시된 작업을 검색 → 응답 생성 대폭 가속.

06

프롬프트 캐싱 규칙

Rules of prompt caching
  • 지속 시간 = 최대 1시간. 활성화는 캐시 브레이크포인트를 수동 추가.
  • 텍스트 블록 포맷: 단축형 content="text"는 캐시 제어 불가. 정식형 [{"type":"text","text":...,"cache_control":{...}}]이 필수.
  • 캐시 범위: 브레이크포인트까지(포함) 모든 콘텐츠. 브레이크포인트 이전 콘텐츠가 바뀌면 전체 캐시 무효화.
  • 처리 순서: 도구 → 시스템 프롬프트 → 메시지.
  • 최대 브레이크포인트 = 요청당 4개. 최소 캐시 임계값 = 1024 토큰 이상.
최적 사용처

반복되는 동일 콘텐츠 — 시스템 프롬프트, 도구 정의, 정적 메시지 접두부.

07

프롬프트 캐싱 실전

Prompt caching in action
  • 도구 스키마 캐싱: 리스트 마지막 도구에 cache_control(type "ephemeral"). 모범 사례: tools 리스트 복사 → 마지막 스키마 복제 → 캐시 제어 추가.
  • 시스템 프롬프트 캐싱: text 블록 딕셔너리로 감싸고 cache_control "ephemeral".
  • 캐시 순서: 도구 → 시스템 프롬프트 → 메시지.
토큰 사용 패턴
필드의미
cache_creation_input_tokens첫 사용 시 캐시에 쓰인 토큰
cache_read_input_tokens후속 동일 요청에서 캐시로부터 검색된 토큰

캐시된 콘텐츠(도구/시스템 프롬프트) 변경 시 무효화 → 새 캐시 생성 강제.

08

코드 실행과 Files API

Code execution and the Files API
Files API

파일을 미리 업로드해 두고 이후 요청에서 파일 ID로 참조(매 요청에 원시 데이터 X). 업로드 → ID 수신 → 이후 ID 사용.

Code Execution

Claude가 격리된 Docker 컨테이너에서 Python 실행. 구현 불필요(스키마만 포함). ⚠️ 컨테이너는 네트워크 접근 불가 → 입출력은 Files API에 의존.

결합 워크플로우

  1. Files API로 파일 업로드 → 파일 ID 획득
  2. 컨테이너 업로드 블록에 ID 포함
  3. Claude에 분석 요청
  4. Claude가 업로드 파일에 접근하며 코드 작성/실행
  5. 분석·결과 반환(생성 파일은 반환된 file ID로 다운로드)
