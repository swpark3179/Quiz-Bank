Introduction to Agent Skills

Skill 생성 · 설정 · 공유 · 문제 해결 + 핵심 개념 도식 + 치트시트

강의 홈 · anthropic.skilljar.com/introduction-to-agent-skills  |  6개 단원 · 예상 학습 시간 약 105분

Skill이란? Claude가 적절한 작업에 자동으로 적용하는 재사용 가능한 마크다운 지침 묶음. 코딩 표준·PR 리뷰 방식·커밋 형식을 매번 설명하는 대신 한 번만 가르치면 된다. 각 Skill은 SKILL.md(이름·설명 frontmatter + 실제 지침)를 담은 폴더다.

이렇게 학습하세요

① 단원 순서대로 정독하고 ② 매칭 흐름 · 우선순위 · progressive disclosure 도식으로 동작 원리를 잡은 뒤 ③ 마지막 치트시트로 메타데이터 필드·커맨드·문제 해결을 빠르게 복습하세요.

01

Skill이란 무엇인가?

What are skills?
학습 목표
  • Skill의 정의와 동작, 저장 위치(개인 vs. 프로젝트) 이해
  • Skill · CLAUDE.md · 슬래시 커맨드의 차이 구분
  • Skill이 적절한 커스터마이징 도구가 되는 상황 식별

핵심 개념

  • Skill은 "한 번만 가르치면" 되는 지침 묶음이다. 작성해 두면 Claude가 관련 상황마다 자동 적용한다.
  • 명령어·리소스가 담긴 폴더이며, 각 Skill은 SKILL.md에 frontmatter(이름·설명) + 그 아래 실제 지침을 둔다.
  • description(설명)이 매칭의 핵심. Claude는 사용자 요청을 모든 Skill의 설명과 비교(시맨틱 매칭)해 활성화한다.
--- name: pr-review description: Reviews pull requests for code quality. Use when reviewing PRs or checking code changes. --- # 아래에 리뷰 체크리스트·형식 선호 등 실제 지침을 작성

SKILL.md frontmatter 예시 — 이름과 설명 아래에 Claude가 따를 실제 지침이 온다.

저장 위치

종류위치특징
개인 Skill~/.claude/skills모든 프로젝트에서 따라다님 (개인 커밋 스타일·문서 형식)
프로젝트 Skill저장소 루트의 .claude/skillsclone하는 모든 팀원이 자동 공유 (팀 표준·브랜드 가이드) · 코드와 함께 커밋

Skill vs. CLAUDE.md vs. 슬래시 커맨드

항상 로드
CLAUDE.md
모든 대화에 항상 포함 (예: "항상 TypeScript strict 사용")
온디맨드
Skill
요청과 일치할 때만 로드. 초기엔 이름·설명만 → 컨텍스트 절약
수동 입력
슬래시 커맨드
사용자가 직접 입력. Skill은 입력 없이 Claude가 자동 인식
핵심 원칙

같은 내용을 Claude에게 반복해서 설명하고 있다면, 그것이 바로 작성해야 할 Skill이다.

02

첫 Skill 만들기

Creating your first skill
학습 목표
  • 올바른 frontmatter 구조로 Skill 생성·테스트·검증
  • Claude가 요청을 Skill에 매칭하는 방식 설명
  • 우선순위 계층(Enterprise·Personal·Project·Plugins) 이해

만드는 과정 (예: PR 설명 작성 Skill)

# ① 디렉터리 생성 (디렉터리명 = Skill명) mkdir -p ~/.claude/skills/pr-description # ② SKILL.md 작성 --- name: pr-description description: Writes pull request descriptions. Use when creating a PR or summarizing changes for a pull request. --- 1. Run `git diff main...HEAD` to see all changes 2. Write a description: ## What / ## Why / ## Changes
  • name: Skill 식별자 · description: 언제 쓸지(매칭 기준) · 두 번째 --- 아래: 활성화 시 따르는 실제 지침.

03

설정과 멀티 파일 Skill

Configuration & multi-file skills
학습 목표
  • allowed-tools, model 등 고급 메타데이터 설정
  • 안정적으로 트리거되는 효과적인 설명 작성
  • progressive disclosure와 멀티 파일 구조로 복잡한 Skill 구성

메타데이터 필드

필드필수설명
name필수소문자·숫자·하이픈만, 최대 64자, 디렉터리명과 일치
description필수언제 사용할지 명시, 최대 1,024자 · 매칭의 핵심
allowed-tools선택Skill 활성 시 사용 가능한 도구 제한
model선택해당 Skill에 사용할 Claude 모델 지정

효과적인 설명 작성법

  • 지침은 명시적으로. 좋은 설명은 두 가지에 답한다 — ① 이 Skill이 무엇을 하는가? ② 언제 사용해야 하는가?
  • 트리거가 안 되면, 사용자가 실제로 요청을 표현하는 방식의 키워드를 추가한다.

allowed-tools로 도구 제한

--- name: codebase-onboarding description: Helps new developers understand how the system works. allowed-tools: Read, Grep, Glob, Bash model: sonnet ---

활성 시 허용된 도구만 사용(편집·쓰기 불가) — 읽기 전용·보안 민감 워크플로우의 가드레일. 생략 시 제한 없음.

Progressive Disclosure (점진적 공개)

Skill은 컨텍스트 윈도우를 대화와 공유한다. 모든 걸 한 파일에 욱여넣으면 컨텍스트 낭비 + 유지보수 곤란. 핵심만 SKILL.md에 두고 상세 자료는 별도 파일로 분리해 필요할 때만 읽게 한다.

scripts/
실행 가능한 코드
references/
추가 문서
assets/
이미지·템플릿·데이터
  • SKILL.md에서 로드 시점을 명시 (예: "시스템 설계 질문 시에만 architecture-guide.md를 읽어라") — 전체 문서가 아니라 목차만 두는 셈.
  • 권장 규칙: SKILL.md는 500줄 이하. 초과 시 reference 파일로 분리.
  • 스크립트는 내용을 컨텍스트에 로드하지 않고 실행 → 출력만 토큰 소비. SKILL.md에 "읽지 말고 실행하라"고 지시. 환경 검증·데이터 변환 등에 유용.
04

Skill vs. 다른 기능

Skills vs. other Claude Code features
학습 목표
  • Skill을 CLAUDE.md · 서브에이전트 · 훅 · MCP 서버와 비교
  • 사용 사례에 맞는 커스터마이징 기능 선택 + 상호 보완적 조합 설계

한눈에 비교

  • CLAUDE.md vs. Skill — 전자는 항상 로드(프로젝트 전역 표준·제약), 후자는 요청 매칭 시 온디맨드(작업별·가끔 필요한 전문 지식).
  • Skill vs. 서브에이전트 — Skill은 현재 대화에 지식을 더함, 서브에이전트는 격리된 컨텍스트에서 독립 수행 후 결과 반환.
  • Skill vs. 훅 — 훅은 이벤트 기반(파일 저장 시 린트 등), Skill은 요청 기반(무엇을 요청하느냐).
  • MCP 서버 — 외부 도구·통합 제공 → Skill과는 완전히 다른 범주.
전형적인 구성 — 각 기능은 자기 전문 영역을 담당한다
기능역할
CLAUDE.md항상 적용되는 프로젝트 표준
Skills온디맨드로 로드되는 작업별 전문성
Hooks이벤트로 트리거되는 자동화 작업
Subagents위임 작업을 위한 격리된 실행 컨텍스트
MCP servers외부 도구·통합
핵심

다른 옵션이 더 맞는데 억지로 Skill에 욱여넣지 말고 여러 기능을 함께 사용하라. Skill은 관련 주제일 때 Claude가 자동 적용해야 할 지식이 있을 때 쓴다.

05

Skill 공유하기

Sharing skills
학습 목표
  • Git 커밋 · 플러그인/마켓플레이스 · enterprise 설정으로 Skill 배포
  • 커스텀 서브에이전트가 특정 Skill을 사용하도록 구성

3가지 공유 방법

  1. 저장소에 커밋 (가장 간단) — .claude/skills에 두면 clone하는 모두가 자동 획득. push → 다음 pull 시 업데이트. .claude엔 agents·hooks·skills·settings가 함께 버전 관리됨. 팀 표준·프로젝트 워크플로우에 적합.
  2. 플러그인으로 배포 — skills 디렉터리에 Skill별 SKILL.md. 마켓플레이스 배포로 발견·설치 가능. 프로젝트 특화도 낮고 두루 유용한 Skill에 적합.
  3. Enterprise 관리형 설정 — 관리자가 조직 전체에 배포, 최고 우선순위(동명 하위 Skill을 덮어씀). 의무 표준·보안·컴플라이언스 — 핵심 키워드는 "반드시(must)".
// 관리형 설정: 플러그인 설치 출처 제어 "strictKnownMarketplaces": [ { "source": "github", "repo": "acme-corp/approved-plugins" }, { "source": "npm", "package": "@acme-corp/compliance-plugins" } ]

Skill과 서브에이전트 — 중요한 함정

  • 서브에이전트는 Skill을 자동으로 보지 못한다. 위임 시 깨끗한 새 컨텍스트로 시작.
  • 내장 에이전트(Explorer·Plan·Verify)는 Skill에 전혀 접근 불가.
  • 커스텀 서브에이전트는 명시적으로 나열할 때만 사용 가능. Skill은 서브에이전트 시작 시 로드(온디맨드 X). frontmatter의 skills 필드에 나열한다.
--- name: frontend-security-accessibility-reviewer tools: Bash, Glob, Grep, Read, WebFetch, Skill... model: sonnet skills: accessibility-audit, performance-check ---

절차: .claude/skills에 Skill 존재 확인 → 새 서브에이전트 생성(/agents) 또는 기존 에이전트에 skills 필드 추가.

06

Skill 문제 해결

Troubleshooting skills
학습 목표
  • 검증기(validator)로 구조적 문제 사전 포착
  • 트리거·로딩·충돌·런타임 문제 진단·수정

문제는 보통 트리거 안 됨 / 로드 안 됨 / 충돌 / 런타임 실패 중 하나다. 가장 먼저 agent skills validator를 실행한다(uv로 설치가 가장 쉬움).

유형별 원인 & 해결

  • 트리거 안 됨 — 원인은 거의 항상 description. 사용자가 실제 말할 트리거 문구를 추가하고 변형("make this faster" 등)으로 테스트.
  • 로드 안 됨 — SKILL.md는 이름 있는 디렉터리 안에, 파일명은 정확히 SKILL.md(대문자 SKILL). claude --debug로 로딩 오류 확인.
  • 잘못된 Skill 사용 — 설명이 너무 비슷함 → 더 구체적·구별되게 작성.
  • 우선순위 충돌 — 동명의 상위(enterprise) Skill이 승리. 이름 변경 또는 관리자와 논의.
  • 런타임 오류 — 의존성 누락(설명에 명시), 권한(chmod +x), 경로(Windows에서도 슬래시 / 사용).
빠른 문제 해결 체크리스트
증상해결
트리거 안 됨설명 개선 + 트리거 문구 추가
로드 안 됨경로·파일명·YAML 문법 확인
잘못된 Skill 사용설명을 서로 더 구별되게
가려짐(shadowed)우선순위 계층 확인 후 이름 변경
플러그인 Skill 누락캐시 삭제 후 재설치
런타임 실패의존성·권한·경로 확인
강의 총정리

Skill을 생성·설정·공유·문제 해결하는 법을 배웠다. 가장 좋은 Skill은 실제 불편함(pain point)에서 나온다 — 가장 자주 반복하는 지침부터 시작하자.
