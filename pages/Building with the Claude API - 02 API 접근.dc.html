01

API 접근하기

Accessing the API

API 접근 흐름 (5단계)

  1. 클라이언트 → 개발자 서버 — 사용자 텍스트를 서버로 전송. ⚠️ 클라이언트 앱에서 Anthropic API를 직접 호출 금지(API 키 보호).
  2. 서버 → Anthropic API — SDK(Python·TS·JS·Go·Ruby) 또는 HTTP. 필수 = API 키 + 모델 + 메시지 리스트 + max_tokens.
  3. 텍스트 생성 — 토큰화 → 임베딩 → 문맥화 → 생성 (아래 도식).
  4. 정지 조건 — max_tokens 도달 또는 end_of_sequence 토큰 생성 시 중지.
  5. 응답 반환 — 생성 텍스트 + 사용량(usage) + stop_reason을 서버 경유로 클라이언트에 표시.

02

요청 보내기

Making a request

설정 4단계

  1. 패키지 설치 — pip install anthropic python-dotenv
  2. API 키 저장 — .env에 ANTHROPIC_API_KEY="your_key" (버전관리 제외)
  3. 환경변수 로드 — python-dotenv로 안전하게 로드
  4. 클라이언트 생성 — anthropic 클라이언트 초기화 + 모델 변수 정의
client.messages.create( model=model, max_tokens=1000, messages=[{"role": "user", "content": "What is quantum computing?"}] )
  • 필수 인자: model, max_tokens, messages. max_tokens는 생성 길이의 안전 상한(목표 길이 아님).
  • 메시지 타입: User({"role":"user","content":...}) · Assistant(모델 생성 응답).
  • 응답 접근: 텍스트만 → message.content[0].text.
03

멀티턴 대화

Multi-turn conversations

⚠️ 핵심 한계: Anthropic API는 어떤 메시지도 저장하지 않는다. 각 요청은 독립적이며 이전 교환의 기억이 없다.

해법

① 코드에서 메시지 리스트를 직접 관리 → ② 후속 요청마다 전체 대화 기록을 함께 전송. 그래야 Claude가 맥락을 유지한다.

흐름 & 헬퍼 함수

사용자 메시지 전송 → 응답 수신 → 응답을 기록에 추가 → 새 메시지 추가 → 전체 기록 전송.

  • add_user_message(messages, text) — 사용자 메시지 추가
  • add_assistant_message(messages, text) — 어시스턴트 응답 추가
  • chat(messages) — 기록을 API로 보내고 응답 반환
04

시스템 프롬프트

System prompts

시스템 프롬프트 = Claude에 역할·행동 패턴을 부여해 응답 스타일·톤을 커스터마이즈. 무엇을 답하는지가 아니라 어떻게 답하는지를 제어한다.

  • 구현: create에 system 키워드 인자로 문자열 전달.
  • 구조: 첫 줄에 역할 지정("You are a patient math tutor") + 이후 구체적 행동 지침.
  • 기술 구현: params 딕셔너리에 프롬프트가 있으면 system 키 조건부 추가, ** 언패킹. None이면 system 제외.
핵심 원칙

시스템 프롬프트는 응답의 접근 방식을 안내한다. 같은 질문도 부여된 역할에 따라 다르게 처리된다(예: 수학 튜터 → 정답 대신 힌트).

05

온도

Temperature

Temperature(0~1) = 토큰 선택 확률에 영향을 주어 생성의 무작위성을 제어. 다음 토큰 선택의 확률 분포를 직접 조작한다.

≈ 0 · 결정적
낮은 온도

항상 최고 확률 토큰 선택. 데이터 추출·일관성이 필요한 사실 작업에.

≈ 1 · 창의적
높은 온도

낮은 확률 토큰 가능성 ↑. 브레인스토밍·글쓰기·마케팅 등에.

높은 값이 항상 다른 출력을 보장하진 않고 변동 확률을 높일 뿐이다.

06

응답 스트리밍

Response streaming

스트리밍 = 전체 응답을 기다리지 않고 생성되는 대로 청크 단위 표시. AI 응답은 10~30초 걸릴 수 있어 즉각적 피드백을 준다.

이벤트 타입
이벤트의미
message_start초기 확인
content_block_start텍스트 생성 시작
content_block_delta실제 텍스트 청크 (가장 중요)
content_block_stop / message_stop생성 완료
  • 기본: client.messages.create(stream=True) → 이벤트 이터레이터.
  • 간소화: client.messages.stream() + text_stream으로 텍스트만 추출.
  • 최종 메시지: stream.get_final_message()로 모든 청크 조립(저장용).
07

모델 출력 제어

Controlling model output

프롬프트 수정 외에 출력을 제어하는 두 기법.

① 프리필링 (Pre-filling)

대화 끝에 어시스턴트 메시지를 수동 추가해 응답 방향 유도. Claude는 그 끝에서 이어서 생성. 예: "Coffee is better because" → 커피 옹호로 이어감.

② 정지 시퀀스 (Stop Sequences)

특정 문자열이 나타나면 즉시 생성 중지. 정지 시퀀스 텍스트는 출력에 미포함. 예: 정지 시퀀스 ", five" → "one, two, three, four".

두 기법 모두 핵심 프롬프트를 바꾸지 않고 응답 방향·길이를 정밀 제어한다.

08

구조화된 데이터

Structured data

문제: Claude는 JSON/코드 생성 시 마크다운·헤더·코멘트를 자동으로 덧붙인다. 사용자는 복붙용 원시 데이터만 원할 때가 많다.

해법 패턴 — 프리필 + 정지 시퀀스

  1. User 메시지 = 구조화 데이터 요청
  2. Assistant 프리필 = 여는 구분자 (예: ```json)
  3. 정지 시퀀스 = 닫는 구분자 (예: ```)
결과

Claude는 프리필을 보고 요청 내용만 생성하고 구분자에서 멈춘다 → 추가 포맷·코멘트 없는 원시 구조화 데이터. JSON·Python 등 모든 구조화 타입에 적용.
