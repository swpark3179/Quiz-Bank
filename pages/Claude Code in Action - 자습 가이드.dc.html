코딩 어시스턴트란?

What is a coding assistant?

핵심 개념

코딩 어시스턴트 = 언어 모델을 사용해 코드를 작성하고 개발 작업을 완료하는 도구.

동작 과정 — 작업을 받으면

STEP 1
작업 받음
→
STEP 2
컨텍스트 수집
→
STEP 3
계획 수립
→
STEP 4
행동 실행

예: 에러 메시지로 버그 수정 → 파일을 읽어 코드베이스 이해 → 해결 계획 → 파일 수정·테스트 실행.

핵심 한계와 해법 — 도구 사용(Tool Use)

한계: 언어 모델은 텍스트 입출력만 처리한다. 직접 파일을 읽거나, 명령을 실행하거나, 외부 시스템과 상호작용할 수 없다. Tool Use 시스템이 이 행동을 대신 수행해 준다.

  1. 어시스턴트가 사용자 요청에 지침을 덧붙인다.
  2. 지침은 행동을 위한 형식화된 응답 형태를 명시한다 (예: "read file: filename").
  3. 언어 모델이 형식화된 행동 요청으로 응답한다.
  4. 어시스턴트가 실제 행동을 실행한다 (파일 읽기, 명령 실행).
  5. 결과를 언어 모델에 다시 전달해 최종 응답을 생성한다.

Claude 모델의 강점

  • 우수한 도구 사용 능력 — 도구 기능을 이해하고 복잡한 작업을 위해 여러 도구를 조합하는 데 뛰어나다.
  • 확장 가능 — 새 도구를 쉽게 추가할 수 있다.
  • 보안 우위 — 코드베이스를 외부 서버로 보내 인덱싱하지 않고 직접 코드를 검색한다.
핵심

모든 언어 모델은 텍스트 생성 외 작업에 도구 사용이 필요하며, 도구 사용의 품질이 코딩 어시스턴트의 효과를 좌우한다.

Claude Code 실전

Claude Code in action

핵심 개념

  • Claude Code = 코드 작업을 위한 도구 기반 AI 어시스턴트.
  • 기본 도구 = 파일 읽기/쓰기, 명령 실행, 기본 개발 작업.

Claude Code 설치

Claude Code setup

로컬에 Claude Code를 설치한다. 전체 안내: code.claude.com/docs/en/quickstart

설치 명령

# macOS, Linux, WSL curl -fsSL https://claude.ai/install.sh | bash # Windows PowerShell irm https://claude.ai/install.ps1 | iex # Windows Command Prompt (cmd.exe) curl -fsSL https://claude.ai/install.cmd -o install.cmd && install.cmd && del install.cmd # macOS (Homebrew) brew install --cask claude-code

운영체제에 맞는 한 줄을 실행한다.

실행 & 인증

  • 설치 후 터미널에서 claude 실행.
  • 최초 실행 시 터미널 색상 테마 선택 + claude.ai 계정 인증 프롬프트가 나타난다.
  • claude를 못 찾거나 네트워크/권한 오류 시 문서의 Troubleshoot installation issues 참고.
  • Amazon Bedrock / Google Cloud Vertex AI / Microsoft Foundry 사용 시 third-party provider 설정 추가 필요.

컨텍스트 추가하기

Adding context

핵심 개념

컨텍스트 관리는 Claude Code 효과의 핵심이다. 관련 없는 정보가 너무 많으면 오히려 성능이 떨어진다.

CLAUDE.md

/init 커맨드 = 최초 실행 시 전체 코드베이스를 분석해 프로젝트 요약·아키텍처·주요 파일이 담긴 CLAUDE.md를 생성한다. 이 파일 내용은 모든 요청에 자동 포함된다.

CLAUDE.md의 세 가지 레벨 (바깥 → 안쪽으로 적용 범위가 좁아짐):

머신(Machine)
모든 프로젝트에 공통으로 적용되는 전역 지침
프로젝트(Project)
팀과 공유 · 소스 컨트롤에 커밋함
로컬(Local)
개인 지침 · 소스 컨트롤에 커밋 안 함

컨텍스트 주입 도구

  • 메모리 모드 (# 기호) — 자연어 요청으로 CLAUDE.md 파일을 똑똑하게 편집한다.
  • @ 기호 — 특정 파일을 멘션해 요청에 포함 → Claude가 검색하게 두지 않고 타겟 컨텍스트를 직접 제공한다.
모범 사례

DB 스키마 같은 핵심 파일을 CLAUDE.md에서 참조해 항상 컨텍스트로 제공한다. 목표는 작업 완수에 딱 필요한 만큼의 관련 정보를 주는 것.

변경 적용하기

Making changes

스크린샷 통합

Control-V(macOS에서도 Command-V 아님)로 스크린샷을 붙여넣어, 수정할 UI 요소를 Claude가 이해하도록 돕는다.

성능 강화 모드 — 넓이 vs 깊이

넓이 · BREADTH
Plan Mode (계획)

Shift + Tab 두 번. 실행 전 더 많은 파일을 조사해 상세 구현 계획을 작성. 다단계·넓은 코드베이스 이해 작업에 유용.

깊이 · DEPTH
Thinking Mode (사고)

"Ultra think" 같은 문구로 발동. 복잡한 로직에 확장된 추론 예산 부여. 까다로운 로직·디버깅에 유용.

두 모드는 조합 가능하며, 둘 다 추가 토큰을 소비하므로 비용을 고려한다.

Git 통합

Claude Code가 변경을 스테이징/커밋하고 설명적 커밋 메시지를 작성할 수 있다.

핵심 워크플로우

문제 영역 스크린샷 → Control-V로 붙여넣기 → 원하는 변경 설명 → (복잡하면) Plan/Thinking 모드 활성화 → 구현 검토 및 수락.

커스텀 커맨드

Custom commands

핵심 개념

커스텀 커맨드 = 슬래시(/)로 접근하는 사용자 정의 자동화 커맨드.

만드는 법

  • 위치: 프로젝트 디렉터리의 .claude/commands/ 폴더.
  • 명명: 파일명이 커맨드명이 됨 (audit.md → /audit).
  • 활성화: 커맨드 파일 생성 후 Claude Code 재시작.
  • 구조: Claude가 실행할 지침이 담긴 마크다운 파일.
  • 인자: 커맨드 텍스트에 $ARGUMENTS 플레이스홀더를 써서 런타임 매개변수(파일 경로, 설명 텍스트 등 임의 문자열)를 받음.

활용

의존성 감사, 테스트 생성, 취약점 수정 같은 반복 작업을 자동화. 실행: /commandname [인자 문자열].

MCP 서버로 확장하기

MCP servers with Claude Code

핵심 개념

  • MCP 서버 = Claude Code 능력을 확장하는 외부 도구. 로컬 또는 원격으로 실행.
  • Playwright MCP 서버 = Claude가 브라우저를 제어해 웹 자동화를 하게 해 주는 인기 서버.

설치 & 권한

claude mcp add [name] [start-command]

권한 관리: 최초 도구 사용 시 승인 필요. settings.local.json의 allow 배열에 mcp__[servername]을 추가하면 자동 승인된다.

실전 예시

Claude가 Playwright로 localhost:3000 탐색 → UI 컴포넌트 생성 → 스타일 품질 분석 → 시각적 피드백을 바탕으로 생성 프롬프트를 자동 개선. 결과적으로 훨씬 나은 컴포넌트 스타일링을 얻는다.

핵심 이점

MCP 서버는 코드 편집을 넘어 외부 시스템이 얽힌 복잡한 다단계 작업(완전한 개발 자동화)을 가능하게 한다.

GitHub 통합

Github integration

핵심 개념

Claude Code GitHub 통합 = Claude를 GitHub Actions 안에서 실행하게 해 주는 공식 통합.

설정 과정

  1. /install-github-app 커맨드 실행
  2. GitHub에 Claude Code 앱 설치
  3. API 키 추가
  4. 자동 생성된 PR이 두 개의 GitHub Actions를 추가

기본 액션

  • 멘션 지원 — 이슈/PR에서 @claude로 작업 할당.
  • PR 리뷰 — 새 PR에 자동 코드 리뷰.

커스터마이징 & 권한

  • .github/workflows 디렉터리의 설정 파일로 액션 커스터마이즈.
  • 커스텀 지침으로 Claude에 컨텍스트/지시 전달, MCP 서버 통합으로 외부 도구(예: Playwright) 접근.
  • 권한: Actions에서 Claude Code의 모든 권한을 명시적으로 나열해야 한다. MCP 서버 도구도 개별 권한 명시 필요(축약 불가).

훅 소개

Introducing hooks

핵심 개념

훅(Hooks) = Claude가 도구를 실행하기 전/후에 실행되는 명령.

Pre-tool use
도구 실행 전 · 검사·차단 가능 · 에러 메시지 전송
→
도구 실행
read · grep · edit · 명령 실행 …
→
Post-tool use
도구 실행 후 · 후속 작업 · Claude에 피드백 제공

설정 & 구조

  • 설정: Claude 설정 파일(global/project/personal)을 수동 편집하거나 /hooks 커맨드로 추가.
  • 구조: 두 섹션(pre/post), 각각 matcher(대상 도구 지정) + 실행할 명령.
  • 예시 용도: 파일 생성 후 자동 포맷, 편집 후 테스트 실행, 파일 접근 차단, 코드 품질 검사, 타입 체크.

훅 정의하기

Defining hooks

훅 구현 절차

  1. 훅 종류 선택 (pre vs post)
  2. 감시할 대상 도구 이름 식별
  3. stdin으로 JSON 형태의 도구 호출 데이터를 받는 명령 작성
  4. tool_name과 입력 매개변수가 담긴 JSON 파싱
  5. 의도를 알리는 종료 코드(exit code)로 종료
종료 코드 (Exit codes)
코드의미
Exit 0도구 호출 허용 (진행)
Exit 2도구 호출 차단 (pre-tool use에서만 가능)
stderr 출력차단 시 Claude에 전달되는 피드백 메시지

도구 호출 데이터: JSON 객체 — tool_name(예: read, grep) + 입력 매개변수(예: file_path).

팁

도구 이름을 외우지 말고 Claude에게 직접 "사용 가능한 도구 목록"을 물어보라.

15

훅 구현하기

Implementing a hook

목표

Claude가 .env 파일 내용을 읽지 못하게 차단하는 훅 구현.

설정 (settings.local.json)

  • 훅 종류: pre-tool use (실행 전 차단)
  • matcher: "read|grep" (파이프 |로 도구 이름 구분)
  • command: "node ./hooks/read_hook.js"

구현 디테일

  • 훅은 stdin으로 JSON 객체 수신: 세션 ID, 도구 이름, 도구 입력, 파일 경로 포함.
  • 로직: 파일 경로에 .env가 포함되면 → console.error()로 에러 로그 + process.exit(2)로 차단.
  • console.error() 출력은 stderr로 가서 Claude 피드백이 된다.
  • tool_input.path로 경로를 확인하되 폴백(fallback) 처리.
핵심 요구사항 & 결과

훅 변경 후에는 Claude Code 재시작 필수. read·grep 양쪽에서 동작하며, .env 접근을 성공적으로 차단하고 그 사실을 사용자/Claude에게 피드백한다.

16

훅의 함정들

Gotchas around hooks
  • 변경 후 재시작 필수 — 훅을 추가·수정하면 Claude Code를 재시작해야 적용된다.
  • 차단은 pre-hook만 — exit 2로 차단 가능한 것은 pre-tool use 훅뿐. post-tool use는 차단 불가.
  • 피드백 경로 — Claude에 메시지를 보내려면 stderr(console.error())로 출력. stdout 아님.
  • 도구 입력 경로 처리 — tool_input.path가 항상 존재한다고 가정하지 말고 폴백 처리.
  • matcher 정확성 — 여러 도구를 노릴 땐 "read|grep"처럼 파이프로 구분, 도구 이름을 정확히 지정.
  • 비용/오버헤드 — 무거운 훅(예: 별도 Claude 인스턴스를 띄우는 훅)은 시간·비용을 늘리므로 핵심 디렉터리에만 적용.

유용한 훅 ② 중복 코드 방지

Another useful hook

문제: Claude가 기존 쿼리/함수를 재사용하지 않고 새로 만든다 — 특히 복잡하게 감싼(wrapped) 작업에서 집중을 잃을 때.

해법: 훅이 쿼리 디렉터리 변경을 감시 → 별도의 Claude 인스턴스를 TypeScript SDK로 기동 → 기존 코드와 비교.

  1. 감시 대상 디렉터리(예: queries/)의 편집 감지
  2. SDK로 새 Claude 인스턴스 실행
  3. 새 코드를 기존 코드와 비교
  4. 중복 발견 시 exit 2 + 피드백
  5. 원래 Claude가 피드백을 받아 기존 코드를 재사용

트레이드오프: 추가 시간·비용 ↔ 더 깨끗한 코드베이스. 권장: 오버헤드 최소화를 위해 핵심 디렉터리만 감시. 두 훅 모두 post-tool-use 패턴으로 Claude의 편집에 즉각적인 피드백·코스 교정을 제공한다.

Claude Code SDK

The Claude Code SDK

핵심 개념

Claude Code SDK = Claude Code를 프로그램적으로 사용하는 인터페이스. CLI · TypeScript · Python 라이브러리 제공. 터미널 버전과 동일한 도구를 포함한다.

주요 특징

  • 주 용도: 더 큰 파이프라인/워크플로우에 통합해 기존 프로세스에 지능을 더함.
  • 기본 권한 = 읽기 전용(파일·디렉터리·grep). 쓰기 권한은 수동 설정 필요 — 쿼리 옵션의 allowedTools 배열에 edit 같은 도구를 지정하거나 .claude 디렉터리 설정으로.
  • 출력: 로컬 Claude Code와 언어 모델 간의 원시 대화(메시지 단위 교환)를 보여 주며, 마지막 메시지가 최종 응답.
적합한 용도

단독 사용보다는 기존 프로젝트 내의 헬퍼 커맨드 · 스크립트 · 훅에 가장 적합하다.

정리 및 다음 단계

Summary and next steps

강의를 마무리하는 단원. (앞에 "Quiz on Claude Code" 퀴즈 단원이 있다.) 핵심을 5가지로 압축하면:

  1. 원리 — Claude Code는 언어 모델 + 도구 사용(tool use)으로 동작하며, 도구 사용의 품질이 어시스턴트의 능력을 좌우한다.
  2. 컨텍스트가 전부 — CLAUDE.md(/init), @(파일 멘션), #(메모리), Plan/Thinking 모드, Escape·compact·clear로 딱 필요한 컨텍스트만 유지한다.
  3. 확장 — 커스텀 커맨드(.claude/commands/), MCP 서버(claude mcp add), GitHub Actions 통합으로 능력을 넓힌다.
  4. 제어와 자동화 — 훅(Hooks)으로 도구 호출 전/후를 가로채(차단은 pre-hook + exit 2, 피드백은 stderr) 자기 교정 피드백 루프를 만든다.
  5. 통합 — Claude Code SDK(CLI/TS/Python)로 파이프라인·스크립트·훅에 Claude Code를 내장한다.
최종 핵심 원칙

Claude Code의 성능은 컨텍스트를 얼마나 잘 관리하고, 도구·훅으로 워크플로우를 얼마나 잘 확장하느냐에 달려 있다.
