에이전트와 워크플로우

병렬화 · 체이닝 · 라우팅 패턴 · 에이전트 설계 · 워크플로우 vs 에이전트

① 시작·개요② API 접근③ 프롬프트 평가④ 엔지니어링⑤ 도구 사용⑥ RAG·검색⑦ Claude 기능⑧ MCP⑨ Anthropic 앱⑩ 에이전트

단일 요청으로 끝나지 않는 작업을 다루는 두 전략 — 워크플로우(정해진 단계)와 에이전트(유연한 계획). 판단 규칙: 단계 순서를 알면 워크플로우, 세부가 불분명하면 에이전트.

01

에이전트와 워크플로우

Agents and workflows

워크플로우 = 단계가 미리 정해진 특정 문제에 대한 일련의 Claude 호출. 예 — 이미지 → 3D 모델 변환기:

↩ 부정확하면 피드백과 함께 모델링부터 반복 — Evaluator-Optimizer 패턴
Evaluator-Optimizer

생산자(Producer)가 출력 생성 → 평가자(Evaluator)가 품질 평가 → 수용할 때까지 루프. 패턴 식별이 자동 구현은 아니며 실제 코드는 직접 작성해야 한다.

병렬화 워크플로우

Parallelization workflows

하나의 복잡한 작업을 여러 동시 서브태스크로 나눈 뒤 결과를 집계(aggregate).

예 — 부품 소재 선택: ❌ 모든 기준을 담은 하나의 큰 프롬프트 / ✅ 소재별 적합성을 평가하는 별도 병렬 요청 → 최종 집계 비교.

이점

집중(각 서브태스크가 한 분석에 전념) · 모듈성(개별 프롬프트 개선/평가) · 확장성(서브태스크 추가) · 품질(복잡한 단일 프롬프트의 혼란 감소).

03

체이닝 워크플로우

Chaining workflows

큰 작업을 하나의 복잡한 프롬프트가 아닌 일련의 순차적 단계로 분할. 각 호출이 하나의 서브태스크에 집중한다.

예: 주제 입력 → 트렌딩 검색 → 흥미로운 것 선택 → 리서치 → 스크립트 작성 → 영상 생성 → SNS 게시.

주 사용처

제약이 많은 긴 프롬프트에서 Claude가 제약을 반복해도 무시할 때(예: "AI 언급 금지, 이모지 금지, 전문적 톤"). 해법: 1단계 초기 출력 수용 → 2단계 발견된 위반을 바탕으로 재작성 요청.

04

라우팅 워크플로우

Routing workflows

사용자 입력을 분류해 적절한 처리 파이프라인을 결정. 첫 요청에서 Claude가 입력을 카테고리로 분류 → 맞춤 프롬프트/도구를 가진 특화 파이프라인으로 라우팅.

  1. 사용자 주제 입력(예: "Python functions")
  2. Claude가 분류(예: "교육적")
  3. 교육 특화 프롬프트 템플릿 사용
  4. 교육적 톤/구조로 스크립트 생성

구조: 하나의 라우팅 단계 → 다수의 특화 파이프라인. 출력이 주제 성격에 맞는다(프로그래밍=교육적, 엔터테인먼트=트렌디).

05

에이전트와 도구

Agents and tools

에이전트 = 제공된 도구로 작업 완료 계획을 스스로 세우는 AI 시스템. 정확한 단계를 모를 때 효과적(워크플로우는 단계를 알 때 더 나음). 가용 도구로 동적 계획.

도구 추상화 원칙

초특화 도구보다 범용/추상 도구를 제공하라. 예: Claude Code는 bash·web_fetch·file_write(추상)를 쓰지 refactor_tool·install_dependencies(특화)를 쓰지 않는다. 에이전트는 소수의 유연한 도구와 가장 잘 작동하며, 도구를 예상 밖 방식으로 창의적으로 조합한다.

06

환경 점검

Environment inspection

환경 점검 = 에이전트가 환경과 행동 결과를 평가해 진행 상황을 이해하고 오류를 처리. 각 행동 후 도구 반환값을 넘어 새 환경 상태를 이해할 피드백이 필요하다.

  • 컴퓨터 사용: 행동(타이핑·클릭)마다 스크린샷으로 변화 확인(결과를 예측 불가하므로).
  • 코드 편집: 파일 수정 전 현재 내용을 읽어 기존 상태 파악.
  • SNS 영상 에이전트: Whisper CPP로 자막·대사 위치 검증, FFmpeg로 구간 스크린샷 점검 → 게시 전 검증.

환경 점검으로 에이전트가 진행을 가늠하고 오류를 감지하며 예상 밖 결과에 적응 — 맹목적으로 작동하지 않는다.

07

워크플로우 vs 에이전트

Workflows vs agents
항목워크플로우에이전트
정의단계가 알려진 사전 정의 호출 연쇄도구를 조합해 미지의 작업을 푸는 유연한 접근
작업 분할작고 특정한 서브태스크 → 집중·정확도↑사전 단계 없이 창의적으로 처리
테스트실행 순서가 알려져 테스트 쉬움실행 경로 예측 불가 → 어려움

성공률구조화로 완료율 높음위임된 복잡성으로 완료율 낮음
권장

신뢰성을 위해 워크플로우 우선. 유연성이 정말 필요할 때만 에이전트. 사용자는 화려한 에이전트보다 100% 작동하는 제품을 원한다. 핵심: 문제를 먼저 신뢰성 있게 풀고, 혁신은 그다음.

전체 강의 총정리

Building with the Claude API는 Anthropic API의 기초부터 에이전트 설계까지를 관통한다.

  1. 기초 — 모델 선택(Opus/Sonnet/Haiku) → API 호출 → 멀티턴(API는 무상태) → 시스템 프롬프트·온도·스트리밍·출력 제어·구조화 데이터.
  2. 품질 — 프롬프트를 eval로 객관 측정하고 명확성·구체성·XML·예시로 개선.
  3. 확장 — 도구 사용(스키마·멀티턴·내장 도구), RAG(청킹·임베딩·BM25·멀티인덱스·리랭킹·컨텍스추얼).
  4. 고급 기능 — 확장 사고, 이미지/PDF, 인용, 프롬프트 캐싱, 코드 실행/Files API.
  5. 생태계 — MCP(도구·리소스·프롬프트), Anthropic 앱(Claude Code·Computer Use).
  6. 설계 — 워크플로우(병렬화·체이닝·라우팅)와 에이전트(추상 도구·환경 점검). 신뢰성 우선, 혁신은 그다음.
관통하는 원칙

측정(eval)하고, 컨텍스트를 정교하게 제어하며, 문제를 신뢰성 있게 푸는 가장 단순한 패턴(워크플로우)부터 선택하라.
