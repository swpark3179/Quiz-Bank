Introduction to Model Context Protocol

MCP 서버·클라이언트 구축 + 3대 프리미티브 도식 + 빠른 복습 치트시트

강의 홈 · anthropic.skilljar.com/introduction-to-model-context-protocol  |  선수 지식: Python · JSON/HTTP 기초

MCP란? Model Context Protocol — Claude에게 컨텍스트와 도구를 제공하는 통신 계층. 서비스마다 도구 스키마/함수를 직접 작성하던 부담을 전용 MCP 서버로 이전한다. 핵심은 도구·리소스·프롬프트 3대 프리미티브다.

이렇게 학습하세요

① 부(Part)별로 정독하고 ② 아키텍처 · 요청–응답 흐름 · 3대 프리미티브 도식으로 구조를 잡은 뒤 ③ 마지막 치트시트로 데코레이터·커맨드·제어 주체를 빠르게 복습하세요.

강의 개요

이 강의에서 배우는 것

  • Python으로 MCP 서버와 클라이언트를 처음부터 구축.
  • 3대 프리미티브 — 도구(tools)·리소스(resources)·프롬프트(prompts) — 로 Claude를 외부 서비스와 연결.
  • 전송 방식 독립적(transport-agnostic) 통신과 클라이언트–서버 메시지 타입.
  • Python SDK 데코레이터로 (수동 JSON 스키마 없이) 서버 작성, 브라우저 인스펙터로 테스트.
  • 각 프리미티브를 언제 쓰는지 구분: 도구(모델 제어)·리소스(앱 제어)·프롬프트(사용자 제어).

예: GitHub API

MCP 클라이언트가 MCP 서버에 연결하고, 서버는 외부 서비스로 향하는 인터페이스 역할을 한다.

어떤 문제를 해결하나

  • 전통적 방식: 서비스를 통합할 때마다 개발자가 도구 스키마와 함수를 직접 작성 → 복잡한 서비스일수록 유지보수 부담 증가.
  • MCP의 해법: 도구 정의·실행을 전용 MCP 서버로 이전. 서버가 기능을 미리 만들어 둔 도구(pre-built tools)로 감싼다.
  • 핵심 가치: 개발자가 직접 작성/유지보수할 필요가 없다. 주로 서비스 제공자가 도구를 작성해 MCP 서버로 패키징한다.

03

MCP 클라이언트

MCP clients

핵심 개념

MCP 클라이언트는 사양(spec)에 정의된 통신 담당자다. 도구를 직접 실행하지 않고, 내 서버와 실제 실행 주체인 MCP 서버 사이의 통신을 중개한다.

주요 메시지 타입

  • list tools 요청/결과 — 사용 가능한 도구 목록 요청 → 서버가 목록으로 응답.
  • call tool 요청/결과 — 인자와 함께 도구 실행 요청 → 서버가 결과 반환.

전형적인 요청–응답 흐름

  1. 사용자 질의(User query) 발생
  2. 서버가 MCP 클라이언트에 도구 목록을 요청
  3. 클라이언트가 MCP 서버에 list tools 요청 전송
  4. 서버가 도구 목록 확보
  5. 서버가 질의 + 도구를 Claude에 전달
  6. Claude가 도구 실행을 요청
  7. 서버가 클라이언트에 도구 실행을 요청
  8. 클라이언트가 MCP 서버에 call tool 요청 전송
  9. MCP 서버가 도구 실행 (예: GitHub API 호출)
  10. 결과가 체인을 거슬러 돌아감 → Claude가 최종 응답 생성 → 사용자에게 전달
핵심

MCP 클라이언트는 중개자다. 도구를 직접 실행하지 않고, 내 서버와 실제 실행 주체인 MCP 서버 간 통신을 매개한다.

클라이언트 구현하기

Implementing a client

핵심 개념

  • MCP 클라이언트: 클라이언트 세션을 감싸 서버 연결 및 리소스 정리(cleanup)를 관리하는 래퍼 클래스.
  • 클라이언트 세션: SDK가 제공하는 실제 연결. 종료 시 connect/cleanup/async enter/async exit로 정리.
  • 목적: 서버 기능을 코드베이스 전반에 노출, 앱 코드와 서버 사이의 인터페이스 제공.
async def list_tools(): result = await self.session.list_tools() return result.tools async def call_tool(tool_name, tool_input): return await self.session.call_tool(tool_name, tool_input)

구현 흐름

  1. 앱이 Claude를 위해 도구 목록을 요청
  2. 클라이언트가 list_tools()로 서버의 도구 확보
  3. Claude가 도구를 선택하고 매개변수 제공
  4. 클라이언트가 call_tool()로 서버에서 실행
  5. 결과를 Claude에 반환
권장 관행

클라이언트 세션을 직접 쓰기보다 더 큰 클래스로 감싸 리소스 관리를 개선한다. client.py를 테스트 하네스로 직접 실행해 연결·도구 목록을 검증.

08

리소스 정의하기

Defining resources

리소스(Resources) = 클라이언트가 읽기(read) 작업으로 데이터를 가져오도록 MCP 서버가 노출하는 기능.

리소스 타입
타입URI 형태예
직접/정적정적 URIdocs://documents
템플릿형와일드카드 매개변수 URIdocuments/{doc_id}

흐름 & 구현

  • 클라이언트가 URI와 함께 read resource 요청 → 서버가 URI를 함수에 매칭·실행 → read resource 결과로 데이터 수신.
  • @mcp.resource 데코레이터 + URI 정의(라우트 같은 주소) + MIME 타입(application/json, text/plain) 설정.
  • 템플릿형은 URI 매개변수가 함수의 키워드 인자가 됨. SDK가 반환값을 문자열로 자동 직렬화.
패턴

구분되는 읽기 작업마다 리소스 하나(목록 조회 vs 단일 항목 조회). MIME 타입은 데이터 형식을 클라이언트에 알려 올바른 역직렬화를 돕는 힌트다.

09

리소스 접근하기

Accessing resources

클라이언트가 서버 리소스에서 데이터를 가져오는 방법.

from pydantic import AnyUrl import json async def read_resource(uri): result = await self.session.read_resource(AnyUrl(uri)) resource = result.contents[0] # 첫 리소스 if resource.mime_type == "application/json": return json.loads(resource.text) # JSON이면 파싱 return resource.text # 그 외 평문

리소스 통합

  • CLI에서 방향키 + 스페이스로 문서 선택.
  • 선택한 리소스 내용이 LLM 프롬프트에 자동 포함 → 문서를 읽기 위한 별도 도구가 불필요.
  • 주요 의존성: json 모듈, 타입 처리용 pydantic.AnyUrl.
10

프롬프트 정의하기

Defining prompts

프롬프트(Prompts) = MCP 서버가 클라이언트에 노출하는 미리 작성·검증된 지침. 클라이언트는 슬래시 커맨드(예: /format)로 접근한다.

구현 패턴

  • @prompt 데코레이터에 name·description 지정, 함수가 인자(예: 문서 ID)를 받음.
  • 메시지 리스트(user/assistant 형식) 반환 → Claude에 직접 전달.
@prompt(name="format", description="rewrites document in markdown") def format_document(doc_id: str) -> list[messages]: return [base.user_message(prompt_text)]

워크플로우: /format → 문서 선택 → 서버가 특화 프롬프트 반환 → 클라이언트가 Claude에 전송 → Claude가 도구로 읽고/재포맷/저장.

핵심 이점

프롬프트 품질을 최종 사용자에게 맡기지 않고, 서버 작성자가 최적화·검증한 프롬프트로 도메인 전문성을 캡슐화한다.

11

클라이언트의 프롬프트

Prompts in the client
async def list_prompts(): result = await self.session.list_prompts() return result.prompts async def get_prompt(prompt_name, arguments): result = await self.session.get_prompt(prompt_name, arguments) return result.messages

인자 흐름 (Arguments flow)

  • 클라이언트 인자 → 프롬프트 함수의 키워드 인자 → 프롬프트 텍스트에 보간(interpolate) → 형식화된 메시지 반환.
  • 반환 형식: AI 모델의 대화 입력을 구성하는 메시지 배열.
  • CLI: /format → 문서 선택 → 문서 ID 포함 프롬프트가 Claude로 전송 → 형식화된 결과 반환.
핵심 개념

프롬프트는 클라이언트가 매개변수와 함께 호출하는 서버 정의 템플릿으로, 동적 콘텐츠 삽입이 가능한 재사용 AI 지침을 제공한다.

MCP 복습 — 3대 프리미티브

MCP review

강의의 핵심. 세 프리미티브는 "누가 제어하느냐"로 구분된다.

도구 Tools
모델 제어
Claude가 언제 실행할지 결정. 모델에 능력을 추가. 예: 계산용 JavaScript 실행.
리소스 Resources
앱 제어
앱 코드가 언제 가져올지 결정. UI 표시·프롬프트 보강용 데이터 주입. 예: 자동완성, Drive 문서 목록.
프롬프트 Prompts
사용자 제어
사용자 행동(버튼·슬래시 커맨드)으로 트리거. 사전 정의된 워크플로우. 예: 채팅 시작 버튼.

제어 패턴으로 용도 판단

  • Claude의 능력이 필요하다 → 도구(tools)
  • 앱의 데이터가 필요하다 → 리소스(resources)
  • 사용자 워크플로우가 필요하다 → 프롬프트(prompts)
강의 총정리 · 핵심 원칙

"이 기능이 누구의 통제를 받아야 하는가?"를 물어보라 — 모델이면 도구, 앱이면 리소스, 사용자면 프롬프트다. 통신은 클라이언트가 list tools/call tool로 중개하고, 세 프리미티브는 데코레이터(@mcp.tool·@mcp.resource·@prompt)로 정의한다.

빠른 복습 — 치트시트

3대 프리미티브 · 제어 주체

  • 도구 Tools모델 제어 → Claude 능력 추가
  • 리소스 Resources앱 제어 → 데이터 주입
  • 프롬프트 Prompts사용자 제어 → 워크플로우

데코레이터 (서버)

  • @mcp.tool도구 + Field 설명
  • @mcp.resourceURI + MIME 타입
  • @promptname·description + 메시지 리스트

클라이언트 메서드

  • list_tools()도구 목록
  • call_tool()도구 실행
  • read_resource()리소스 읽기 (AnyUrl)
  • list/get_prompt()프롬프트 목록·취득

커맨드 & 실행

  • 실행uv run main.py
