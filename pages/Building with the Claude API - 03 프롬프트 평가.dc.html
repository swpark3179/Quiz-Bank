프롬프트 평가(eval) = 객관적 지표로 프롬프트의 효과를 측정하는 자동화 테스트. 엔지니어들은 흔히 프롬프트를 과소 테스트한다 — 반복·배포 전에 평가 파이프라인으로 객관적 성능 점수를 얻어라.

02

전형적인 Eval 워크플로우

A typical eval workflow
  1. 초기 프롬프트 초안 — 최적화할 베이스라인
  2. 평가 데이터셋 생성 — 테스트 입력 모음(3개~수천 개, 손수 또는 LLM 생성)
  3. 프롬프트 변형 생성 — 각 입력을 프롬프트 템플릿에 보간(interpolate)
  4. LLM 응답 수집 — 각 변형을 Claude에 넣어 출력 수집
  5. 응답 채점 — 점수화(예: 1~10), 평균으로 전체 성능 산출
  6. 반복(Iterate) — 점수 기반 수정 후 전체 반복, 버전 비교
핵심

표준 방법론은 없다. 간단한 커스텀 구현으로 시작해도 된다 — 객관적 점수가 A/B 비교를 통한 체계적 개선을 가능케 한다.

03

테스트 데이터셋 생성

Generating test datasets
  • 구조: 사용자 요청을 기술하는 task 속성을 가진 JSON 객체 배열.
  • 생성 방식: 수동 조립 또는 Claude로 자동화(생성엔 Haiku 같은 빠른 모델).

생성 과정

  1. Claude에 테스트 케이스 생성 요청
  2. 어시스턴트 ```json 프리필 사용
  3. 정지 시퀀스 ``` 설정
  4. 응답을 JSON으로 파싱
  5. 파일(dataset.json)로 저장
04

Eval 실행하기

Running the eval

Eval 실행 = 테스트 케이스를 프롬프트와 병합 → LLM 실행 → 출력 채점.

3개의 핵심 함수
함수역할
run_prompt테스트 케이스를 프롬프트와 병합해 Claude에 전송, 출력 반환
run_test_caserun_prompt 호출 + 결과 채점, 요약 딕셔너리 반환
run_eval데이터셋을 순회하며 각각 run_test_case 호출, 결과 조립
Eval 파이프라인 핵심

데이터셋 + 프롬프트 + LLM + 채점기(grader). 초기 버전(v1)은 하드코딩 점수(score=10)의 한계 → 다음 단계는 제대로 된 채점 시스템으로 교체.

모델 기반 채점

Model based grading
세 가지 채점기 유형
유형설명
코드 채점기프로그래밍적 검사(길이, 단어 존재, 구문 검증, 가독성)
모델 채점기원본 출력을 평가하는 추가 API 호출 — 품질·지침 준수 평가에 매우 유연
인간 채점기사람이 평가 — 가장 유연하나 느리고 번거로움
모델 채점기 구현

점수만 요청하지 말 것 — 강점/약점/근거/점수를 함께 요청(점수만 요청하면 중간값으로 치우침). JSON 포맷 + 프리필 + 정지 시퀀스로 점수·근거 파싱, 평균을 최종 지표로.

06

코드 기반 채점

Code based grading

코드·JSON·정규식을 담은 LLM 출력에 대한 자동 검증.

  • validate_json() — 파싱 시도 → 유효 10, 에러 0
  • validate_python() — AST 파싱 시도 → 유효 10, 에러 0
  • validate_regex() — 정규식 컴파일 시도 → 유효 10, 에러 0

데이터셋에 기대 출력 타입을 지정하는 "format" 키 포함(JSON/Python/RegEx). 모델엔 주석 없이 순수 코드/JSON/정규식만 응답하도록 지시.

점수 시스템

최종 점수 = (모델 점수 + 구문 점수) / 2 — 의미적 평가 + 구문 검증 결합. 한계: 적절한 검증기 선택을 위해 기대 포맷을 미리 알아야 한다.
