## 문제
프롬프트 엔지니어링에 대한 설명으로 가장 적절한 것은?
1. AI 모델을 처음부터 학습시키는 기술이다
2. AI에게 효과적으로 질문하여 원하는 답변을 얻는 기술이다
3. AI 모델의 파라미터를 조정하는 기술이다
4. AI 모델을 배포하는 기술이다
5. AI 모델의 속도를 최적화하는 기술이다

## 정답
2

## 해설
1. **오답** - 모델 학습은 Pre-training이나 Fine-tuning에 해당합니다.
2. **정답** - 프롬프트 엔지니어링은 "AI에게 질문 잘하는 기술"입니다. 어떻게 질문하느냐에 따라 답변의 품질이 달라지므로, 효과적인 프롬프트 작성법을 연구합니다.
3. **오답** - 파라미터 조정은 Fine-tuning 영역입니다.
4. **오답** - 모델 배포는 MLOps 영역입니다.
5. **오답** - 속도 최적화는 양자화, Pruning 등의 영역입니다.

## 문제
프롬프트 엔지니어링의 변화에 대한 설명으로 올바른 것은?
1. 2025년에는 프롬프트 엔지니어링이 완전히 불필요해졌다
2. 2023년에는 중요하지 않았으나 2025년에 갑자기 중요해졌다
3. 2023년에는 특정 직군의 전문 역량이었으나, 2025년에는 모든 사람이 갖춰야 할 역량으로 변화했다
4. 프롬프트 엔지니어링은 변화 없이 동일하게 유지되고 있다
5. 최신 모델에서는 프롬프트와 관계없이 항상 동일한 결과가 나온다

## 정답
3

## 해설
1. **오답** - 여전히 고급 프롬프트 기술은 중요합니다.
2. **오답** - 2023년 ChatGPT 등장 시 프롬프트 엔지니어가 연봉 4억원으로 주목받을 정도로 중요했습니다.
3. **정답** - 2023년에는 전문 직종으로 각광받았으나, 최신 LLM들이 똑똑해지면서 2025년에는 특정 직군이 아닌 모든 사람이 갖춰야 할 기본 역량으로 변화했습니다.
4. **오답** - LLM의 발전에 따라 프롬프트 엔지니어링도 변화하고 있습니다.
5. **오답** - 프롬프트에 따라 결과가 달라지는 것은 여전합니다.

## 문제
Google Gemini 팀이 제시한 좋은 프롬프트의 4가지 요소가 **아닌** 것은?
1. Persona (역할)
2. Task (과제)
3. Context (맥락)
4. Format (형식)
5. Temperature (온도)

## 정답
5

## 해설
1. **4요소에 포함** - Persona는 AI에게 특정 역할을 부여합니다. 예: "너는 10년 경력의 마케팅 전문가야"
2. **4요소에 포함** - Task는 무엇을 해야 하는지 명확히 합니다. 예: "신제품 런칭 전략을 만들어줘"
3. **4요소에 포함** - Context는 배경 정보를 제공합니다. 예: "우리 회사는 스타트업이고, 예산은 1천만원이야"
4. **4요소에 포함** - Format은 원하는 출력 형태입니다. 예: "표로 정리해줘"
5. **정답(미포함)** - Temperature는 API 파라미터로, 프롬프트 구성요소가 아닙니다.

1. 역할 부여는 모든 상황에서 항상 성능을 크게 향상시킨다

3. 역할 부여는 답변 형식에는 유효하지만, 단순 지식 질문에서는 큰 효과가 없을 수 있다
4. 역할 부여는 최신 모델에서만 작동한다
5. 역할 부여는 영어 프롬프트에서만 효과가 있다

3

4. **오답** - 모델 버전과 직접적인 관련은 없습니다.
5. **오답** - 언어와 직접적인 관련은 없습니다.

## 문제
프롬프트 구조화에서 사용되는 특수 문자의 역할로 **틀린** 것은?
1. `#` - 제목/섹션 구분
2. `---` - 구역 분리
3. `[]`, `<>` - 변수나 특정 항목 표시
4. `**` - 중요 내용 강조
5. `@@` - LLM 내부 명령어 호출

## 정답
5

## 해설
1. **옳음** - `#`은 마크다운에서 제목이나 섹션을 구분하는 데 사용됩니다.
2. **옳음** - `---`는 구역을 분리하는 데 사용됩니다.
3. **옳음** - `[]`, `<>`는 변수나 특정 항목을 표시하는 데 사용됩니다.
4. **옳음** - `**`는 중요 내용을 강조하는 데 사용됩니다.
5. **틀림(정답)** - `@@`는 프롬프트 구조화에서 사용되는 특수 문자가 아닙니다. LLM 내부 명령어도 존재하지 않습니다.

## 문제
Prompt Chaining에 대한 설명으로 올바른 것은?
1. 하나의 긴 프롬프트로 모든 작업을 한 번에 처리하는 기법이다
2. 복잡한 작업을 여러 단계로 나누어 순차적으로 진행하는 기법이다
3. 여러 LLM을 동시에 호출하는 기법이다
4. 프롬프트를 암호화하는 기법이다
5. LLM의 출력을 다른 프로그래밍 언어로 변환하는 기법이다

## 정답
2

## 해설
1. **오답** - Prompt Chaining은 오히려 작업을 나누는 기법입니다.
2. **정답** - Prompt Chaining은 LLM이 이전 대화를 기억하는 특성을 활용하여, 복잡한 작업을 여러 단계로 나누어 진행합니다. 이를 통해 Context 길이 단축, 정확도 향상, 토큰 비용 절감 효과를 얻습니다.
3. **오답** - 여러 LLM 동시 호출은 Multi-Agent와 관련됩니다.
4. **오답** - 암호화와는 무관합니다.
5. **오답** - 언어 변환과는 무관합니다.

## 문제
Prompt Chaining의 장점으로 **해당하지 않는** 것은?
1. Context 길이 단축
2. 정확도 향상
3. 토큰 비용 절감
4. 모델 파라미터 자동 업데이트
5. 단계별 오류 확인 가능

## 정답
4

## 해설
1. **장점에 해당** - 한 번에 모든 정보를 넣지 않아도 됩니다.
2. **장점에 해당** - 단계별로 처리하여 실수가 감소합니다.
3. **장점에 해당** - 필요한 정보만 주고받아 효율적입니다.
4. **정답(해당 안 함)** - Prompt Chaining은 모델 파라미터를 업데이트하지 않습니다. 파라미터 업데이트는 Fine-tuning에서 이루어집니다.
5. **장점에 해당** - 각 단계의 결과를 확인할 수 있습니다.

## 문제
Few-Shot Prompting에 대한 설명으로 올바른 것은?
1. 예시 없이 규칙만 설명하는 기법이다
2. LLM에게 몇 가지 예시를 보여주고 그 패턴을 따르게 하는 기법이다
3. 대량의 데이터로 모델을 학습시키는 기법이다
4. 프롬프트의 길이를 최소화하는 기법이다
5. 오직 영어로만 사용 가능한 기법이다

## 정답
2

## 해설
1. **오답** - 예시 없이 규칙만 설명하는 것은 Zero-Shot에 가깝습니다.
2. **정답** - Few-Shot Prompting은 LLM에게 몇 가지 예시를 보여주고, 그 패턴을 인식하고 따라하게 하는 기법입니다. 규칙을 설명하는 것보다 예시를 보여주는 것이 더 효과적인 경우가 많습니다.
3. **오답** - 대량 데이터 학습은 Pre-training입니다.
4. **오답** - Few-Shot은 예시를 포함하므로 프롬프트가 길어질 수 있습니다.
5. **오답** - 모든 언어에서 사용 가능합니다.

## 문제
Few-Shot Prompting 사용 시 주의해야 할 점으로 올바른 것은?
1. 예시는 많을수록 항상 좋다
2. 예시의 스타일, 길이, 어조가 결과에 영향을 미친다
3. 예시는 모두 동일한 형식이어야 한다
4. 예시는 정답만 포함해야 하며 오답 예시는 금지된다
5. 예시와 실제 질문은 완전히 다른 주제여야 한다

## 정답
2

## 해설
1. **오답** - 너무 많은 예시는 오히려 컨텍스트를 낭비할 수 있습니다.
2. **정답** - 예시의 스타일, 길이, 어조가 결과에 직접적인 영향을 미칩니다. 짧은 예시는 짧은 결과를, 긴 예시는 긴 결과를 유도합니다. 예시의 다양성도 일반화 능력에 영향을 줍니다.
3. **오답** - 다양한 케이스의 예시가 일반화 능력을 향상시킵니다.
4. **오답** - 상황에 따라 오답 예시도 유용할 수 있습니다.
5. **오답** - 예시와 실제 질문은 관련 주제여야 패턴 학습이 가능합니다.

## 문제
Chain-of-Thought(CoT) Prompting의 핵심 아이디어로 가장 적절한 것은?
1. 답을 최대한 빠르게 생성하도록 유도한다
2. 답을 바로 말하지 않고 풀이 과정을 보여주도록 요청하여 정답률을 높인다
3. 여러 모델의 답변을 종합한다
4. 프롬프트를 최대한 짧게 작성한다
5. 모델의 학습 데이터를 변경한다

## 정답
2

## 해설
1. **오답** - CoT는 속도보다 정확도에 초점을 맞춥니다.
2. **정답** - CoT는 LLM에게 "바로 답하지 말고, 풀이 과정을 보여줘"라고 요청하는 기법입니다. 중간 과정을 글로 쓰면서 "생각"하면 더 정확한 결과를 낼 수 있습니다.
3. **오답** - 여러 모델 종합은 앙상블 기법입니다.
4. **오답** - CoT는 오히려 출력이 길어집니다.
5. **오답** - 학습 데이터 변경은 Fine-tuning입니다.

CoT 논문(2022)이 현대 LLM 발전에 기여한 핵심 인사이트는?

4. 영어로만 학습해야 한다

2

3. **오답** - CoT는 오히려 출력을 길게 유도합니다.
4. **오답** - 다국어 학습도 중요합니다.

## 문제
CoT의 잠재적 문제점인 "Over-Thinking"에 대한 설명으로 올바른 것은?
1. 모델이 너무 빨리 답변하는 문제이다
2. 모델이 너무 많이 생각해서 오히려 성능이 떨어지는 문제이다
3. 모델이 전혀 생각하지 않는 문제이다
4. 프롬프트가 너무 짧아서 생기는 문제이다

2

3. **오답** - 전혀 생각하지 않는 것의 반대 문제입니다.
4. **오답** - 프롬프트 길이와는 직접적 관련이 없습니다.
5. **오답** - 하드웨어 문제가 아닙니다.

3. **오답** - 간결하라는 지시는 CoT의 목적과 맞지 않습니다.

## 문제
Step-Back Prompting의 핵심 아이디어로 가장 적절한 것은?
1. 답을 바로 만들지 않고, 먼저 필요한 구조와 요소를 파악한다
2. 최대한 빠르게 답변을 생성한다
3. 프롬프트를 최소화한다
4. 여러 모델의 답변을 비교한다
5. 이전 답변을 삭제하고 다시 시작한다

## 정답
1

## 해설
1. **정답** - Step-Back Prompting은 "답을 바로 만들지 말고, 먼저 필요한 구조와 요소를 파악하자"는 아이디어입니다. 한 단계 물러난 질문(Stepback Question)으로 추상적 답변을 얻은 후, 본격적인 추론을 수행합니다.
2. **오답** - 빠른 답변 생성이 목적이 아닙니다.
3. **오답** - 프롬프트 최소화와는 무관합니다.
4. **오답** - 여러 모델 비교는 다른 기법입니다.
5. **오답** - 이전 답변 삭제와는 다릅니다.

## 문제
Step-Back Prompting의 활용 예시로 올바른 것은?
1. 코딩 작업 시 바로 코드를 작성하도록 요청한다
2. 보고서 작성 시 먼저 목차와 각 섹션 개요를 요청한다
3. 모든 질문에 한 문장으로 답변하도록 요청한다
4. 예시를 여러 개 제공한다
5. 모델을 변경한다

## 정답
2

## 해설
1. **오답** - Step-Back은 바로 코드 작성이 아니라 먼저 구조를 파악합니다.
2. **정답** - Step-Back Prompting의 좋은 예는 보고서 작성 시 "보고서의 목차와 각 섹션 개요를 먼저 알려줘"라고 요청하는 것입니다. 제안서 평가도 "좋은 제안서의 평가 기준을 먼저 만들어줘"처럼 적용할 수 있습니다.
3. **오답** - 한 문장 답변은 Step-Back과 무관합니다.
4. **오답** - 예시 제공은 Few-Shot입니다.
5. **오답** - 모델 변경은 Step-Back과 무관합니다.

2. 모델 자체가 Tool Calling을 네이티브로 지원하게 되었다
3. Tool Calling은 더 이상 사용되지 않는다
4. 프롬프트 기법으로만 Tool Calling이 가능하다
5. 오픈소스 모델만 Tool Calling을 지원한다

## 정답
2

## 해설
1. **오답** - Tool Calling은 더욱 발전했습니다.
2. **정답** - 2024년 이전에는 ReAct, Reflexion 등 프롬프트 기법으로 Agent를 구현했지만, 2024-2025년에는 Kimi K2, GPT-5, Claude 등 모델 자체가 Tool Calling을 네이티브로 지원합니다.
3. **오답** - Tool Calling은 더욱 활발히 사용됩니다.
4. **오답** - 네이티브 지원으로 프롬프트 기법이 덜 필요해졌습니다.
5. **오답** - 상용 모델도 지원합니다.

## 문제
Agent 기술의 발전 연대기 순서로 올바른 것은?
1. Agent Q → Reflexion → ReAct
2. Reflexion → ReAct → Agent Q
3. ReAct → Agent Q → Reflexion
4. ReAct → Reflexion → Agent Q
5. Agent Q → ReAct → Reflexion

## 정답
4

## 해설
1. **오답** - ReAct가 가장 먼저입니다.
2. **오답** - ReAct가 Reflexion보다 먼저입니다.
3. **오답** - Reflexion이 Agent Q보다 먼저입니다.
4. **정답** - 발전 순서: ReAct(2022, 추론과 행동의 결합) → Reflexion(2023, 자기 반성 메커니즘) → Agent Q(2024, MCTS + 강화학습)입니다.
5. **오답** - ReAct가 가장 먼저입니다.

## 문제
소형 오픈소스 모델(sLLM)에서 Tool Calling 한계를 해결하는 방법으로 올바른 것은?
1. 더 큰 GPU를 사용한다
2. Tool 수를 제한하고 단순화하거나, Prompt로 형식을 안내한다
3. Tool을 완전히 제거한다
4. 모델을 처음부터 다시 학습시킨다
5. 영어로만 사용한다

## 정답
2

## 해설
1. **오답** - GPU 크기가 직접적인 해결책은 아닙니다.
2. **정답** - sLLM의 Tool Calling 한계 해결책: ① Tool Message를 모델이 이해 못하면 Prompt로 형식 안내, ② Tool이 많으면 필요한 Tool만 제공하여 Context 과부하 방지, ③ 복잡한 Tool 연결이 불가하면 Tool 수 제한 및 단순화합니다.
3. **오답** - Tool 제거는 해결책이 아닙니다.
4. **오답** - 재학습은 비용이 많이 듭니다.
5. **오답** - 언어가 주요 해결책은 아닙니다.
