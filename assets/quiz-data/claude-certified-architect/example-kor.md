## 문제 1
### **웹 검색 서브에이전트가 복잡한 주제를 검색하던 중 시간 초과(timeout) 오류가 발생했습니다. 이 오류 정보가 코디네이터 에이전트로 어떻게 전달될지 설계해야 합니다. 지능적인 복구를 가장 효과적으로 지원하는 오류 전파 방식은 무엇일까요?**

| 보기 | 설명 |
|------|------|
| 1. 하위 에이전트가 재시도 로직을 자체적으로 처리한다. | |
| 2. 타임아웃을 성공으로 위장하여 빈 결과를 반환한다. | |
| 3. 예외를 최상위로 직접 전달하여 전체 워크플로우를 종료한다. | |
| **4. 오류 유형, 시도한 쿼리, 부분 결과, 가능한 대체 접근 방식을 포함한 구조화된 오류 컨텍스트를 코디네이터에게 반환한다.** | |

**정답: 4.**

**해설:**

다중 에이전트 시스템에서 오류 전파의 핵심 원칙은 **"코디네이터가 충분한 정보를 바탕으로 복구 전략을 결정할 수 있어야 한다"** 는 것입니다.

1. **오답** - 하위 에이전트가 재시도 로직을 자체 처리하면 코디네이터가 상황을 알지 못해 제어권을 잃습니다.
2. **오답** - 타임아웃을 성공으로 위장해 빈 결과를 반환하면 오류가 은폐되어 시스템 신뢰성이 훼손됩니다.
3. **오답** - 예외를 최상위로 직접 전달하면 단일 오류가 전체 워크플로우를 중단시킵니다.
4. **정답** - 구조화된 오류 컨텍스트를 전달하면 코디네이터가 재시도·대안 검색·부분 결과 활용 등 최적의 복구 전략을 선택할 수 있습니다.

## 문제 2
### **웹 검색 하위 에이전트는 요청된 5개 소스 범주 중 3개에 대해서만 결과를 반환했고, 문서 분석 하위 에이전트는 제공된 모든 문서를 성공적으로 처리했습니다. 이제 종합(synthesis) 하위 에이전트가 이렇게 품질이 혼재된 입력 자료를 바탕으로 결과 요약을 생성해야 합니다. 가장 효과적인 오류 전파 전략은 무엇일까요?**

| 보기 | 설명 |
|------|------|
| 1. 성공한 소스의 결과만 사용한다. | |
| 2. 누락된 소스의 재시도를 요청한 뒤 합성을 지연시킨다. | |
| 3. 불완전한 데이터를 오류로 처리한다. | |
| **4. 종합 결과 출력에 커버리지(coverage) 주석을 포함하여, 어떤 결과가 잘 뒷받침되는지와 어떤 주제 영역에 자료 부족으로 인한 공백이 있는지를 구조화하여 표시한다.** | |

**정답: 4.**

**해설:**

부분 성공 상황에서는 결과를 숨기거나 실패로 처리하는 것이 아니라, 어떤 부분이 신뢰 가능하고 어떤 부분이 불완전한지를 투명하게 표시해야 합니다.

1. **오답** - 성공한 소스만 사용하면 누락된 데이터를 표시하지 않아 정보를 은폐하게 됩니다.
2. **오답** - 재시도를 요청한 뒤 합성을 지연하면 불필요한 대기와 복잡성만 늘어납니다.
3. **오답** - 불완전한 데이터를 오류로 처리하면 전체 재시도를 유발하는 과잉 반응입니다.
4. **정답** - 커버리지 주석으로 신뢰도와 공백을 명확히 표시하면 합성 에이전트가 투명한 결과를 제공할 수 있습니다.

## 문제 3
### **"AI가 창조 산업에 미치는 영향"이라는 주제로 시스템을 실행했더니, 최종 보고서에 시각 예술 분야만 포함되고 음악·작문·영화 제작은 누락되었습니다. 코디네이터 로그를 확인하니 주제가 "디지털 예술 창작에서의 AI", "그래픽 디자인에서의 AI", "사진에서의 AI"라는 세 가지 하위 작업으로 분해되어 있었습니다. 이 현상의 가장 유력한 원인은 무엇일까요?**

| 보기 | 설명 |
|------|------|
| **1. 코디네이터의 작업 분해가 너무 좁아, 주제의 모든 관련 영역을 포괄하지 못하는 하위 작업으로 배정되었다.** | |
| 2. 문서 분석 에이전트가 지나치게 엄격한 관련성 기준으로 비(非)시각 창작 분야 소스를 걸러냈다. | |
| 3. 웹 검색 에이전트의 쿼리가 충분히 포괄적이지 않아 더 많은 분야를 다루도록 확장해야 한다. | |
| 4. 합성(종합) 에이전트에게 받은 결과에서 커버리지 공백을 식별하는 지침이 없다. | |

**정답: 1.**

**해설:**

코디네이터 로그가 주제를 "디지털 아트·그래픽 디자인·사진"(모두 시각 예술)으로만 분해하고 음악·작문·영화 제작을 완전히 누락했음을 직접 보여줍니다. 하위 에이전트들은 배정받은 작업을 모두 올바르게 수행했으므로, 좁은 작업 분해가 명백한 근본 원인입니다.

1. **정답** - 코디네이터가 주제를 시각 예술 3개로만 분해해 음악·작문·영화가 처음부터 배정에서 빠졌습니다. 좁은 작업 분해가 누락의 근본 원인입니다.
2. **오답** - 비시각 소스는 애초에 배정 자체에 없었으므로 문서 분석의 필터링 문제가 아닙니다.
3. **오답** - 웹 검색 쿼리도 배정된(시각 예술) 범위 안에서는 정상 동작했습니다. 쿼리 확장이 아니라 분해 범위가 문제입니다.
4. **오답** - 합성 단계에서 공백을 표시하면 증상 보고에는 도움이 되지만, 데이터가 애초에 수집되지 않은 근본 원인은 코디네이터의 분해에 있습니다.

## 문제 4
### **문서 분석 하위 에이전트가 구문 분석할 수 없는 손상된 PDF 파일을 만났습니다. 시스템의 오류 처리를 설계할 때 이러한 오류를 처리하는 가장 효과적인 방법은 무엇일까요?**

| 보기 | 설명 |
|------|------|
| 1. 손상된 파일을 무시하고 넘어간다. | |
| **2. 오류와 관련된 컨텍스트를 코디네이터 에이전트에 반환하여, 코디네이터가 어떻게 처리할지 결정하도록 한다.** | |

2. **정답** - 오류 컨텍스트를 코디네이터에 전달하면 코디네이터가 최적의 대응을 결정할 수 있습니다.

## 문제 7
### **운영 로그를 분석한 결과, "업로드한 분기 보고서를 분석해 주세요"라는 요청이 문서 분석 에이전트 대신 웹 검색 에이전트로 45% 확률로 라우팅되는 일관된 패턴이 나타났습니다. 도구 정의를 보면 웹 검색 에이전트는 `analyze_content`("콘텐츠를 분석하고 핵심 정보를 추출하는") 도구를, 문서 분석 에이전트는 `analyze_document`("문서를 분석하고 핵심 정보를 추출하는") 도구를 가지고 있습니다. 이 잘못된 라우팅 문제를 어떻게 해결해야 할까요?**

| 보기 | 설명 |
|------|------|
| **1. 웹 검색 도구의 이름을 `extract_web_results`로 변경하고, 설명을 "웹 검색·URL에서 가져온 정보를 처리하고 반환한다"로 갱신한다.** | |
| 2. 코디네이터 안내 메시지에 올바른 라우팅을 보여주는 몇 가지 예시(few-shot)를 추가한다. | |
| 3. 업로드 파일인지 웹 콘텐츠인지 판별하는 사전 라우팅 분류기를 추가한다. | |
| 4. 웹 검색 도구는 그대로 두고 문서 분석 도구의 설명만 확장한다. | |

**정답: 1.**

**해설:**

두 도구의 이름과 설명이 거의 동일합니다(`analyze_content` vs `analyze_document`, 둘 다 "extracts key information"). 근본 원인은 이 모호한 도구 정의이므로, 웹 검색 도구의 이름·설명을 명확히 구별해 의미적 겹침을 제거해야 합니다.

1. **정답** - 웹 검색 도구를 `extract_web_results`로 이름 변경하고 설명을 웹 검색·URL로 한정하면, 두 도구 간 의미적 겹침이 사라져 코디네이터가 문서 분석과 웹 검색을 명확히 구별합니다. 근본 원인을 직접 해결합니다.
2. **오답** - few-shot 예시는 증상을 다루는 우회책일 뿐, 모호한 도구 정의는 그대로 남아 다양한 표현에 일반화되지 못합니다.
3. **오답** - 사전 분류기는 또 하나의 컴포넌트를 더하는 복잡한 해법으로, 도구 정의를 명확히 하면 불필요합니다.
4. **오답** - 문서 도구 설명만 확장하면 웹 도구의 모호한 `analyze_content`("extracts key information")가 그대로 남아 겹침이 해소되지 않습니다.

## 문제 8
### **웹 검색 에이전트(85,000토큰)와 문서 분석 에이전트(70,000토큰)의 출력값 합계가 155,000토큰이지만, 합성 에이전트는 50,000토큰 미만이 입력될 때 최적 성능을 보입니다. 가장 효과적인 해결책은 무엇일까요?**

| 보기 | 설명 |
|------|------|
| **1. 상류(웹 검색·문서 분석) 에이전트가 장황한 서술과 추론 대신 구조화된 데이터(핵심 사실, 인용문, 관련성 점수)를 반환하도록 수정한다.** | |
| 2. 출력을 벡터 DB에 저장한 뒤 검색하여 사용한다. | |
| 3. 중간 요약 도구를 추가한다. | |
| 4. 배치(batch)로 순차 처리한다. | |

**정답: 1.**

**해설:**

컨텍스트 윈도우 제한 문제는 하위 에이전트의 출력 형식을 압축·구조화하는 것이 근본적인 해결책입니다.
